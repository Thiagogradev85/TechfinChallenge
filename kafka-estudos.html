1. Os containers Docker do projeto

Quando você roda docker-compose up -d na raiz do projeto, 6 containers sobem. Cada um tem uma responsabilidade:

🦒 techfin-zookeeper

Porta interna: 2181

Gerenciador de coordenação do Kafka. Guarda metadados: quais brokers existem, quais topics, quais partitions pertencem a qual broker. O Kafka precisa dele para funcionar. Versões novas do Kafka (KRaft) eliminam o Zookeeper, mas ainda é comum no mercado.

📨 techfin-kafka

Porta externa: 29092 | Porta interna: 9092

O broker Kafka propriamente dito. Recebe mensagens dos Producers, armazena nos topics/partitions e entrega aos Consumers. É o coração da mensageria. Sua API usa localhost:29092 para se conectar.

🖥️ techfin-kafka-ui

Porta: 8080 → http://localhost:8080

Interface visual para o Kafka. Permite ver topics, mensagens, offsets, consumer groups e criar/deletar topics sem usar linha de comando. Equivalente ao pgAdmin, mas para o Kafka.

🐰 techfin-rabbitmq

AMQP: 5672 | UI: 15672 → http://localhost:15672

Broker de mensageria alternativo. O projeto suporta RabbitMQ e Kafka — você troca pelo frontend em Configurações. RabbitMQ é mais simples, Kafka é mais escalável.

🐘 techfin-postgres

Porta: 5432

Banco de dados PostgreSQL. Armazena usuários, clientes e transações. Os dados persistem porque usa um volume Docker (postgres_data) — sobrevive a reinicializações.

🔧 techfin-pgadmin

Porta: 5050 → http://localhost:5050

Interface visual para o PostgreSQL. Login: admin@techfin.com / admin123. Ao conectar, use postgres como host (nome do container), não localhost.

Por que duas portas no Kafka?

9092 = comunicação interna entre containers Docker. 29092 = comunicação externa (sua API .NET no host). Sua API usa localhost:29092 porque roda fora do Docker.

2. Comandos Docker essenciais

Gerenciar os containers

# Subir todos os containers em background
docker-compose up -d

# Subir apenas containers específicos
docker-compose up -d postgres pgadmin

# Ver status de todos os containers
docker ps

# Ver logs de um container específico
docker logs techfin-kafka

# Parar tudo (dados ficam salvos)
docker-compose down

# Parar E apagar tudo (reset total — perde dados do banco)
docker-compose down -v

Comandos Kafka dentro do container

# Listar todos os topics
docker exec -it techfin-kafka kafka-topics --bootstrap-server localhost:9092 --list

# Criar topic com 3 partitions
docker exec -it techfin-kafka kafka-topics --bootstrap-server localhost:9092 --create --topic transacao.aprovada --partitions 3 --replication-factor 1

# Ver detalhes de um topic (partitions, replicas)
docker exec -it techfin-kafka kafka-topics --bootstrap-server localhost:9092 --describe --topic transacao.aprovada

# Deletar topic
docker exec -it techfin-kafka kafka-topics --bootstrap-server localhost:9092 --delete --topic transacao.aprovada

# Ler mensagens de um topic em tempo real (consumer de linha de comando)
docker exec -it techfin-kafka kafka-console-consumer --bootstrap-server localhost:9092 --topic transacao.aprovada --from-beginning

4. Conceitos fundamentais do Kafka

📦
Topic
Canal nomeado onde mensagens são publicadas. É como uma tabela no banco — tem um nome e armazena dados. No projeto: transacao.aprovada. Producers publicam nele, Consumers leem dele.
🛣️
Partition
Divisão física de um topic. Pense nas faixas de uma rodovia — cada mensagem vai para uma faixa específica e fica lá na ordem em que chegou. Mais partitions = mais paralelismo. No Módulo 2 criamos 3 partitions.
📍
Offset
Posição de uma mensagem dentro de uma partition. Começa em 0, nunca diminui, nunca reutiliza. O Consumer guarda qual offset já processou — se cair e voltar, continua de onde parou. No log você viu: Offset: 2.
📤
Producer
Quem publica mensagens no topic. No projeto: KafkaPublisher na Transacao API. Ele serializa o evento para JSON e envia com uma key (o ClienteId) que determina a partition.
📥
Consumer
Quem lê mensagens do topic. No projeto: KafkaConsumer na Clientes API, rodando como BackgroundService. Ele fica em loop infinito consumindo mensagens e commita o offset após processar.
👥
Consumer Group
Grupo de consumers que trabalham juntos. Cada mensagem vai para apenas um consumer do grupo. No projeto: group id = clientes-api. Se você subir 2 instâncias da Clientes API, elas dividem as partitions entre si.
🔑
Message Key
Campo opcional da mensagem que determina a partition via hash. Mesma key = mesma partition = ordem garantida. No projeto usamos o ClienteId como key, garantindo que todas as transações de um cliente sejam processadas em ordem.
✅
Commit de Offset
Ato de informar ao Kafka que uma mensagem foi processada com sucesso. No projeto usamos commit manual (EnableAutoCommit = false) — só commitamos após o handler executar. Se a API cair antes do commit, a mensagem será reprocessada.

Por que EnableAutoCommit = false?

// KafkaConsumer.cs
var config = new ConsumerConfig
{
    EnableAutoCommit = false,  // commit manual — mais seguro
};

// Só commita DEPOIS de processar com sucesso
handler.HandleAsync(evento).GetAwaiter().GetResult();
consumer.Commit(result);  // ← aqui
Por que isso importa

Com AutoCommit, o Kafka marcaria a mensagem como lida antes de você terminar de processar. Se a API cair no meio, a mensagem seria perdida. Com commit manual: processou → commitou. Falhou → não commitou → reprocessa.

Módulo 2 — Em andamento
Partições — ordenação e paralelismo

Como partitions funcionam

Topic: transacao.aprovada (3 partitions)
Partition 0
débito R$1000 João
crédito R$200 João
débito R$500 João
Partition 1
débito R$300 Maria
crédito R$100 Maria
Partition 2
débito R$700 Pedro

Verde = João → sempre Partition 0 (mesma key = mesmo hash = mesma partition)

A fórmula do roteamento

// O Kafka usa internamente:
partition = hash(key) % total_partitions

// Exemplo com ClienteId do projeto:
hash("38490a9b-3b99-48b3-8a33-4519fdf344cd") % 3 = 0
// Toda transação desse cliente sempre vai para Partition 0

// No KafkaPublisher.cs:
Key = evento.ClienteId  // ← essa linha garante a ordenação por cliente

Por que ordem importa em fintech

Cenário sem key (round-robin)
T=1: débito  R$9000  → Partition 2  (offset 0)
T=2: crédito R$100   → Partition 0  (offset 0)
T=3: débito  R$200   → Partition 1  (offset 0)

// Consumer processa Partition 0 primeiro:
// crédito R$100 processado ANTES do débito R$9000
// Limite: 1000 + 100 = 1100 → depois -9000 → NEGATIVO INVÁLIDO
Cenário com key = ClienteId
T=1: débito  R$9000  → Partition 0  (offset 0)
T=2: crédito R$100   → Partition 0  (offset 1)  ← mesma key, mesma partition
T=3: débito  R$200   → Partition 0  (offset 2)

// Consumer processa na ordem garantida:
// -9000 → +100 → -200 = ordem correta sempre

Consumer Groups — escalabilidade e processamento independente

O que é um Consumer Group

Um Consumer Group é um grupo de consumers que dividem o trabalho de consumir um topic. Cada partition é atribuída a apenas um consumer do grupo por vez.

Com 1 instância — pega tudo

Topic: transacao.aprovada (3 partitions)

Consumer Group: clientes-api
  Partition 0 ──→ KafkaConsumer (instância única)
  Partition 1 ──→ KafkaConsumer (instância única)
  Partition 2 ──→ KafkaConsumer (instância única)

Com 2 instâncias — divide automaticamente

Consumer Group: clientes-api
  Partition 0 ──→ KafkaConsumer A (instância 1)
  Partition 1 ──→ KafkaConsumer B (instância 2)
  Partition 2 ──→ KafkaConsumer A (instância 1)

// Kafka redistribui as partitions automaticamente (rebalance)
// Mais instâncias que partitions = instâncias extras ficam ociosas
// Por isso partitions devem ser >= instâncias esperadas
Regra importante

Número máximo de consumers úteis em um grupo = número de partitions. Se você tem 3 partitions e 5 instâncias, 2 ficam ociosas. Por isso dimensionar partitions é uma decisão de arquitetura.

Dois grupos = dois processamentos independentes

Topic: transacao.aprovada
         │
         ├──→ grupo clientes-api   → debita/credita limite no banco
         │         offset próprio: já leu até offset 4
         │
         └──→ grupo auditoria-api  → loga para auditoria/compliance
                   offset próprio: já leu até offset 2 (mais lento, tudo bem)

A mesma mensagem é entregue para os dois grupos. Cada grupo tem seu próprio offset salvo no topic interno __consumer_offsets — um não interfere no outro. Você pode ter quantos grupos quiser consumindo o mesmo topic.

Caso de uso real — XP Investimentos

grupo: limite-api

Debita/credita o limite do cliente em tempo real. Precisa ser rápido — SLA de 100ms.

grupo: auditoria-api

Grava log de auditoria para compliance. Pode ser mais lento — SLA de segundos.

grupo: notificacao-api

Envia push notification ao cliente. Independente dos outros dois.

grupo: analytics-api

Alimenta dashboard de BI com dados de transações. Processa em batch.

O que você viu no Kafka UI — aba Consumers

Group ID: clientes-api
  Topic: transacao.aprovada
  Partition 0 → Consumer host: ... | Offset: 1 | Lag: 0
  Partition 1 → Consumer host: ... | Offset: 0 | Lag: 0
  Partition 2 → Consumer host: ... | Offset: 5 | Lag: 0

// Lag = mensagens não processadas (Next Offset - Committed Offset)
// Lag = 0 significa que o consumer está em dia
// Lag alto = consumer está atrasado — alerta de monitoramento em produção

O que acontece se a Clientes API cair

// Transacao API continua publicando normalmente
offset 5: débito João  → Partition 2  ✓ salvo no Kafka
offset 6: crédito Maria → Partition 0  ✓ salvo no Kafka
offset 7: débito Pedro  → Partition 2  ✓ salvo no Kafka

// Clientes API cai — consumer para
// Lag sobe: 3 mensagens pendentes

// Clientes API volta — consumer reinicia do último offset commitado
// Processa offset 5, 6, 7 em ordem — nenhuma mensagem perdida
Isso é o poder do Kafka

RabbitMQ perderia as mensagens se o consumer não estivesse online. O Kafka guarda tudo pelo período de retenção configurado (30 dias no nosso projeto) — o consumer pode cair e voltar quando quiser.

Retry e Dead Letter Queue — o que fazer quando uma mensagem falha

O problema — mensagem travada para sempre

No KafkaConsumer atual, se o handler lançar uma exceção o offset não é commitado. O consumer tenta reprocessar a mesma mensagem infinitamente, bloqueando todas as mensagens seguintes da mesma partition.

// KafkaConsumer.cs — comportamento atual
catch (Exception ex)
{
    _logger.LogError(ex, "Erro ao processar — continuando...");
    // offset NÃO commitado → mesma mensagem na próxima iteração
    // Partition 2 fica bloqueada enquanto offset 7 falhar
}

// Resultado:
// offset 7 → falha → tenta de novo → falha → tenta de novo → ∞
// offset 8, 9, 10... nunca são processados

O que é Dead Letter Queue (DLQ)

Após N tentativas falhas, a mensagem é movida para um topic separado em vez de bloquear. O time de operações investiga as mensagens no DLQ sem impactar o fluxo principal.

transacao.aprovada
→
tenta 1x → falha
→
tenta 2x → falha
→
tenta 3x → falha
→
transacao.falha (DLQ)

Implementação no projeto

// KafkaConsumer.cs — com retry e DLQ
var tentativas = 0;
const int maxTentativas = 3;

while (!stoppingToken.IsCancellationRequested)
{
    var result = consumer.Consume(stoppingToken);
    tentativas = 0;
    var processado = false;

    while (tentativas < maxTentativas && !processado)
    {
        try
        {
            tentativas++;
            handler.HandleAsync(evento).GetAwaiter().GetResult();
            consumer.Commit(result);      // sucesso → commita
            processado = true;
        }
        catch (Exception ex)
        {
            _logger.LogWarning("Tentativa {T}/{Max} falhou", tentativas, maxTentativas);

            if (tentativas >= maxTentativas)
            {
                // esgotou tentativas → manda pro DLQ
                dlqProducer.ProduceAsync("transacao.falha", result.Message);
                consumer.Commit(result);  // commita para não reprocessar
                _logger.LogError("Mensagem enviada para DLQ após {Max} tentativas", maxTentativas);
            }
        }
    }
}

At-Least-Once vs Exactly-Once

GarantiaO que significaRisco
At-Least-OnceMensagem processada uma ou mais vezesProcessamento duplicado se cair após processar mas antes de commitar
At-Most-OnceMensagem processada zero ou uma vezPerda de mensagem se cair antes de processar
Exactly-OnceExatamente uma vez, garantidoComplexo, requer transações Kafka + idempotência
Padrão do mercado

At-Least-Once + idempotência no handler é o padrão mais usado. O handler verifica se já processou aquela mensagem antes de agir — se sim, ignora. Mais simples que Exactly-Once e igualmente seguro.

Cenários de falha comuns em fintech

Banco fora do ar

Handler não consegue salvar o novo limite. Com retry: tenta 3x aguardando o banco voltar. Com DLQ: registra para reprocessar manualmente quando o banco voltar.

Cliente não existe

Evento chegou mas o ClienteId não está no banco. Não adianta retry — é erro de dados. Vai direto pro DLQ para investigação.

Timeout de rede

Chamada HTTP para outra API demorou demais. Retry com backoff exponencial (espera 1s, 2s, 4s) resolve na maioria dos casos.

Mensagem corrompida

JSON inválido, campo faltando, tipo errado. Deserialização falha. Vai pro DLQ imediatamente — retry não vai resolver.

Como ver no Kafka UI

Após implementar o DLQ, crie o topic transacao.falha com 1 partition no Kafka UI. Quando uma mensagem falhar 3x ela aparecerá lá na aba Messages para investigação.

Múltiplos Topics — quando separar e como um serviço é produtor e consumidor ao mesmo tempo

O problema de um topic genérico

No projeto temos um único topic transacao.aprovada para todas as transações. Funciona, mas mistura eventos com propósitos diferentes. À medida que o sistema cresce, aparecem perguntas:

Diferentes consumidores

O módulo de antifraude quer só débitos. O de cashback quer só créditos. Com um topic único, ambos consomem tudo e descartam metade.

Diferentes retenções

Eventos de auditoria precisam ficar 1 ano. Eventos de heartbeat precisam ficar 1 hora. Um topic só não permite isso.

Diferentes volumes

Crédito tem 10x mais volume que débito. Com topics separados você configura mais partições para crédito sem afetar débito.

Convenção de nomes de topics

O padrão do mercado é dominio.entidade.acao no formato snake_case ou kebab-case:

// Padrão: domínio.entidade.ação
transacao.aprovada          // atual — genérico
transacao.debito.aprovado   // mais específico por tipo
transacao.credito.aprovado

cliente.limite.atualizado   // evento downstream — após o handler atualizar o limite
cliente.criado
cliente.bloqueado

pagamento.pix.processado
pagamento.ted.falhou

Um serviço pode ser produtor E consumidor

Esse é o padrão mais importante do Módulo 5. No projeto atual, a Clientes API só consome. No mundo real ela também produziria eventos após processar — criando uma cadeia de eventos:

→ publica em →
transacao.aprovada

Clientes API como produtor e consumidor

A Clientes API consome transacao.aprovada e produz cliente.limite.atualizado. Ela é os dois ao mesmo tempo — é assim que sistemas event-driven se compõem sem acoplamento direto.

Subscrevendo em múltiplos topics

O Confluent.Kafka permite passar uma lista para Subscribe. O consumer recebe mensagens de todos os topics da lista e você identifica de qual veio por result.Topic:

// Subscribe aceita uma lista — consumer recebe de todos os topics
consumer.Subscribe(new[] { "transacao.debito", "transacao.credito" });

// No loop, identifique de qual topic veio a mensagem
var result = consumer.Consume(stoppingToken);

if (result.Topic == "transacao.debito")
{
    // lógica específica de débito
    handler.HandleDebitoAsync(evento).GetAwaiter().GetResult();
}
else if (result.Topic == "transacao.credito")
{
    // lógica específica de crédito
    handler.HandleCreditoAsync(evento).GetAwaiter().GetResult();
}

Publicar em topics diferentes no Producer

O producer já escolhe o topic em cada chamada — basta mudar a string:

// KafkaPublisher — roteamento por tipo de transação
public async Task PublicarAsync(TransacaoAprovadaEvent evento)
{
    var topicName = evento.Tipo == "debito"
        ? "transacao.debito"
        : "transacao.credito";

    await _producer.ProduceAsync(topicName, new Message<string, string>
    {
        Key = evento.ClienteId,
        Value = JsonSerializer.Serialize(evento)
    });
}

// Ou, após processar, publicar um evento downstream
await _dlqOrNotificationProducer.ProduceAsync("cliente.limite.atualizado", new Message<string, string>
{
    Key = clienteId,
    Value = JsonSerializer.Serialize(new { clienteId, novoLimite, timestamp = DateTime.UtcNow })
});

Quando criar um novo topic vs reutilizar

Crie um novo topic se...Reutilize o topic se...
Consumidores diferentes com necessidades diferentesTodos os consumidores precisam do mesmo dado
Retenção ou configuração diferenteA lógica de diferenciação é simples (um campo tipo)
O evento tem semântica claramente diferenteO volume é baixo e não justifica a complexidade
Times diferentes consomem de forma independenteVocê está no início e quer simplicidade
Armadilha comum — topic explosion

Criar um topic para cada variação pequena gera centenas de topics difíceis de gerenciar. No Kafka a regra de ouro é: um topic por tipo de evento de negócio, não por variação técnica.

No projeto: o que temos e o que poderíamos ter

// Atual — um topic, tipo no payload

// Alternativa — topics separados por tipo

// Event chain — Clientes API também produz

  └→ Clientes API consome → atualiza limite
